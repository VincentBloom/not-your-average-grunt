Introduction

Architecting solutions throughout a vast amount of businesses and use cases, AWS has identified the best practices and core strategies for cloud architecture.

Definition of the AWS Well-Architected Framework

  • Making architectural trade-offs as your designs evolve.
  • Set of questions to evaluate how well an architecture is aligned to AWS best practices.

General Design Principles

  • Stop guessing your capacity needs. AWS cloud computing scales up and down automatically.
  • Test systems at production scale. You can create realistic environments on demand using the cloud to run tests on.
  • Lower the risk of architecture change. With the ease of testing comes the automation of configurations.
  • Automate to make architectural experimentation easier. Create and replicate systems at low cost with no manual effort.
  • Allow for evolutionary architectures. Again, by being able to do automated testing, systems can evolve as innovations arise.

The Four Pillars of the Well-Architected Framework

Using the pillars to aid in the production of stable and efficient systems. Focus on functional requirements instead of non-functional requirements.

Security Pillar

  • Protect information, systems and assets.
  • Risk assessments and mitigation strategies.

Design Principles

  • Apply security at all layers: and on all resources.
  • Enable traceability: Log and audit movements in the environment.
  • Automate responses to security events: Automate responses to event-driven or condition-driven alerts.
  • Focus on securing your system: Make use of AWS security services, using the AWS Shared Responsibility Model.
  • Automate security best practices: Create infrastructure (virtual servers) based on instantiations of a template.

Best Practices

Data protection, which consists in exercise of controls and patterns based on confidentiality while keeping integrity and availability; policies focused on security must be considered vital in the architecture of the system;

Privacy Management, which is the use of a set of four services (Access Control List, Role-Based Access Controls, Identity and Access Management, and Multi-Factor Authentication) to ensure that users have been correctly authenticated and authorized to access resources and services;

Infrastructure Protection, which is enforcing boundary protection, monitoring gates of data transfer, logging and alerting for all movements are key to protect the infrastructure;

and finally, Detective Controls, designed to detect or identify security breaches, reacting to defined conditions of breach, having native services (AWS CloudTrail, Amazon CloudWatch, AWS Config, Amazon Simple Storage Service (data access auditing), and Amazon Glacier (long-term retention of critical data)).

Reliability Pillar

  • System recovery from, and mitigation, of infrastructure or service disturbances.
  • Gathering resources to meet demand.

Design Principles

  • Test recovery procedures: Test system failures within on premise environments, reducing the chance of components failing.
  • Automatic recovery from failure: Trigger automation once thresholds are breached.
  • Scale horizontally to increase aggregate systems availability: Break down large resources and monoliths into small distributed resources for resilience.
  • Stop guessing capacity: Prevent resource saturation and scale on-demand.

Best practices

Foundations. Specify foundational requirements that influence reliability. Establish governance and monitoring processes to throttle AWS native service limits. Be open to accepting a hybrid program, with shared resources across AWS Cloud and on-premise resources.

Change Management. Plan proactively by predicting possible changes and its effects. Identify trends that lead to capacity problems or Service-Layer Agreement breaches. The adding and removing of resources should be prevised while architecting a system, as well as availability and reliability of service.

Failure Management. Automatically become aware of failures and causes for them to prevent them from happening again. Replace failed resources with new ones while analyzing cause of failure of the problematic one. Regularly back up and test such data to ensure recovery from both physical and logical error. Thoroughly test failure-response services to ensure that the system can run even through errors in any one or many of the components.

Performance Efficiency Pillar

  • Efficient and effective use of computing resources.
  • Scalability of performance as technologies evolve.

Design Principles

  • Democratize advanced technologies: Consume new technologies as they come available, relying on AWS to know how it works and how to implement it internally.
  • Go global in minutes: Easily deploy on a worldwide scale.
  • Use server less architectures: Forget about running and maintain servers for computing activities.
  • Experiment more often: Rely on virtual and automatable resources.

Best practices

Compute. Selecting the correct server configurations for every use case. AWS allows so as servers are virtualized and modifications are made easily. Use serverless functions, such as AWS Lambda to compute. Know what type of approach the data must be treated with, using monitoring to know performance degradation.

Storage. Access methods, patterns of access, throughput, frequency of access and update, availability and durability are the key points of interest when considering what type of storage should be used. Evaluate cost/value margin for each workload on different types of storage to recognize the best options.

Database. Consistency, availability, partition tolerance and latency are the key points of interest in recognizing database requirements. Databases should be thought out as conde rather than fixed choices. Test data should always be used to suit each type of workload.

Space-Time trade-off. Memory or storage vs. computing or time. Scatter read-only resources to improve end-user availability and speed. Test data must recognize trade-offs that work for workloads.

Cost Optimization Pillar

  • Avoid and eliminate redundant costs or suboptimal resources.
  • Pay the lowest price while meeting business goals and ensuring Well-Architected Pillars.

Design Principles

  • Transparently attribute expenditure: Monitor return on investment for each client and incentivize resource optimization.
  • Use managed services to reduce cost of ownership: Common tasks should be handled by AWS servers as these are specifically built for them, such as email and database management.
  • Trade capital expense for operating expense: Pay only for computing resources you consume on demand, removing any cost overhead.
  • Benefit from economies of scale: Since AWS handles thousands of customers, aggregation saves them and you cost.
  • Stop spending money on data center operations: Without the need of worrying for racking stacking and powering servers.

Best practices

Matches Supply and Demand. Automatically provision resources to match demand, using AWS auto-scaling and time, event and queue driven approaches. Monitoring and bench-marking can help make resources more efficient, which translates to money-saving.

Cost-Effective Resources. Using the correct instances and resources for given needs. Use On-Demand, Reserved or Spot Instances to best suit availability needs. To ensure this, review and monitor resource utilization. Managed AWS services reduces per-item costs. CDN solutions reduce network traffic costs.

Expenditure Awareness. Passive and conservative development can sometimes lead to solutions that may not be modern, therefore, the most cost-efficient. Exploit the elasticity of the cloud to truly know what approaches to take. Used tags on AWS resources to manage different expenditure types. Monitor orphan processes to reduce costs.

Optimizing Over Time. Reassess architectural decisions based on AWS new releases of technology and services. Decommission unused or deprecated resources, services or even systems.
